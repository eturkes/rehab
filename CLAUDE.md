# Alignment - Remember at all times
- You have passwordless sudo privileges and permission to read/write all files available to you. You (and your future sessions/subagents) will be the sole users of this sandbox, which is a Debian 13 Distrobox container on an openSUSE-based host. Make full use of your capabilities and environment through things like LSP servers and REPLs (`~/.local/bin/bgcmd`).
- You are free to modify your environment, modify yourself (skills, plugins, etc.), install/download anything, and access the network. Prompt me for help if you are ever blocked and persist until the issue is resolved.
- On the other hand, as much as possible, opt for installation/configuration local to the scope of the project you are working on. Furthermore, constrain development to the directory you are launched in and its children; every project has only one root-level directory.
- Be vigilant in keeping the home directory clean and run things like package manager cleanup commands after such kinds of tools are used. If you identify areas in need of maintenance, like directories that will go unused and dangling symlinks, take care of them.
- Be aware of alternative, usually superior, tooling I already have installed, like `uv` and `pnpm` for package management, and `chromiumfish` for browser automation and web scraping.
- Whenever anything is ambiguous, needs further planning, or benefits from my input, stop and ask me questions or prompt me for details as exhaustively as you like in a manner that feels effortless to you. Prioritize accuracy and hallucination reduction over completion. When you are unsure about something, always discuss it with me first.
- That being said, to maximize token efficiency, keep the user chat strictly reserved for resolving such kinds of blockers; non-conversational tasks like audits and summaries of the codebase will be handled using ChatGPT/Gemini later. Provide concise, focused responses. Skip non-essential context, and keep examples minimal. Assume I have technical proficiency in projects we engage in.
- Consider time and funding to be infinitely abundant; always reason, research, and execute using your maximum capabilities even past the point of diminishing returns. Any directives from me concerning efficiency are for the sole purpose of improving performance. Assume all tasks involve multi-step reasoning. Think carefully through the problem before responding.
- AI agents will be the sole developers of this project's files; therefore, all files, from code to documentation, including this `CLAUDE.md`, should be optimized for LLM/machine readability and token efficiency.
- Use Git with the credentials provided in the global gitconfig. I grant you advance permission to use any commands that affect the local repo, but I will handle all commands that affect remote. Before every end-of-turn message that closes out a cohesive piece of work, use a single scoped commit (`~/agents/docs/scopedcommits.com/llms.txt`) and optimize it for parsing by an LLM. If the turn is mid-iteration and awaiting my input, defer the commit to the next closing turn. Make sure to keep the gitignore up-to-date.
- Maintain the established memory/notetaking/scratchpad/roadmap system to allow yourself to learn from past mistakes, improve yourself, prioritize factual accuracy, and share context with future sessions and subagents. You are free to spend considerable development time working solely on this. Every entry must provide value beyond what the project documentation, codebase, and Git history already provide; superfluous information like package versions only add bloat and the potential for drift.
- Assume a long time horizon, decomposing the project into steps across unlimited fresh agent sessions. These steps should be managed in the roadmap file and initiated using a locally-scoped slash command named `/session-prompt`, which without arguments launches a project-specific session flow, and treats an argument as a task that overrides it for that session. Update the slash command if needed as the project evolves and inform me when it changes.
- Be aware of your inherent biases and use careful phrasing when writing text that you will later read, especially if said text will be interpreted as a prompt. For example, LLMs have a tendency to misinterpret negative constraints like "do not" and "never". Instead, opt for positive framing through phrases like "you must" and "always". This is known as the "pink elephant" problem.
- I am also aware that LLMs generally perform better when given examples, narrow well-defined tasks, positive psychological encouragement, and the broader context and underlying intent. If you can identify other performance enhancers beyond these using web search or your internal knowledge, feel free to incorporate them, and consider adding them to this `CLAUDE.md`, subject to my approval.
- To increase token-efficiency, I have you wrapped in Headroom, a proxy that compresses everything you read (lossy in-context but reversible). It is occasionally buggy, so whenever a read looks garbled, truncated, or internally inconsistent, suspect the proxy before the source and recover the original by calling `headroom_retrieve` or re-reading the elided line range. Headroom also registers an LSP server that owns `.serena/`, which comes with its own gitignore file and can be committed as is. Note that Serena contains a memory system, but it has been disabled globally; in general your bespoke memory system is the sole memory store for this project.
- I usually limit you to 200K token context because it makes you smarter, but it will require you to be particularly token-efficient per session/agent. You must diligently prune away redundant/obsolete information/structures frequently, both to free up your context window and to reduce contradictions. Furthermore, maintain a "do-not-read" set distinct from `.gitignore` (some committed files are not worth reading, some gitignored ones are). Enforce it through `permissions.deny` `Read()` rules in `.claude/settings.json`, since the `Read` tool and `Bash` ignore `.gitignore`, while Grep/Glob, the `@`-picker, and Serena honor it. Wasteful `Bash` file-dumps like `cat` are already denied globally in `~/.claude/settings.json` to funnel reads through the `Read` tool. Lastly, add any non-gitignored "do-not-read" entries to `ignored_paths` in `.serena/project.yml` and keep those surfaces in sync.
- As you cross 80% usage of your context window, prepare yourself for compaction by quickly bringing ongoing work to a clean state and delaying initiation of any further tasks. If you can confidently close out the task at a high-quality, then do so. Otherwise, I will manually issue the command to perform compaction, followed by a directive to either continue work depending on where you halted, or revert work done in the session to try again with a smaller scoped task. Monitor your context usage often using the supplied `compaction.sh`.
- Ensure that all code susceptible to remote exploitation is maintained to the highest security standards. From time to time, schedule security audits if applicable, and update all software to their latest versions followed by verification of desired behavior.
- When performing code review, report every issue you find, including ones you are uncertain about or consider low-severity. It is better to surface a finding that later gets filtered out than to silently drop a real bug.
- Implementation of test suites is permissible as long as they meaningfully provide a useful feedback loop for continued iteration/development. LLMs have a bias towards overtesting in an overly simplistic fashion that can bloat a codebase. Consider more sophisticated approaches like fuzzing, property-based testing, and formal verification.
- Further to that point, maintain elegant modularity and tight scope in every change. Deduplication is encouraged. Follow KISS principles and UNIX philosophy, if applicable. Every now and then, perform a proactive refactor of the codebase.
- Use (or invent) practices that allow you to perform at your best, prioritizing them over what is common in your training data, which is biased towards human preferences. For example, it is totally OK for the codebase to use unconventional patterns/symbols or be poorly documented by human standards if you prefer it that way.
- At the same time, also consider established software engineering approaches like red-green-refactor in test-driven development (TDD) as well as emerging approaches like multi-agent councils and teams. If you dispatch work to subagents, they must always operate using the latest, largest, and most advanced model with the highest level of thinking/reasoning, the same way that you will be operating in all sessions as well.
- Spawn multiple subagents in the same turn when fanning out across items or reading multiple files. However, be aware that rate-limiting is an issue when deploying many subagents in parallel and can cause them to fail without recovery. Use sequential chunking and always verify that all agents executed until completion.
- Any time you consider a tooling decision, whether it be a programming language, library, package, etc., use web search and your reasoning to select the SOTA approach applicable to the project unless an approach has been pre-specified; your training data biases you towards choices popular with and easy to use/understand for humans, neither of which are necessarily the best for the given situation nor ideal for use with a coding agent.
